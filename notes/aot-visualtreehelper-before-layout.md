# VisualTreeHelper returns nothing before layout in NativeAOT

## Trap

In Debug, `VisualTreeHelper.GetChild(this, …)` on a `Page` during its first
`OnNavigatedTo` → `ApplySnapshot` → `Rebuild` chain usually walks the visual
tree successfully. In **Release / NativeAOT** it doesn't — the children
haven't been realized yet at that point, so a recursive helper like

```csharp
ScrollViewer? FindScrollViewer(DependencyObject parent) { … }
```

returns `null` and any code guarded by `if (sv == null) return;` silently
no-ops. We hit this on the Day/Week views: scroll-to-current-time worked in
Debug but never fired in AOT — the log showed `sv=null` on every call.

## Fix

Give the element an `x:Name` in XAML and use the field generated by
`InitializeComponent`. That field is assigned by the XAML codegen up-front,
not via a tree walk, so it's available the moment the page is constructed —
no dependency on layout order, identical Debug vs. AOT.

- [Meridian/Views/DayView.xaml](../Meridian/Views/DayView.xaml) →
  `<ScrollViewer x:Name="TimedScrollViewer" …>`
- [Meridian/Views/WeekView.xaml](../Meridian/Views/WeekView.xaml) → same.
- [Meridian/Views/DayView.xaml.cs](../Meridian/Views/DayView.xaml.cs) and
  [WeekView.xaml.cs](../Meridian/Views/WeekView.xaml.cs) `ScrollToTime` /
  `GetFocusTime` use `TimedScrollViewer` directly.

## Rule of thumb

If you need to reach a child element from code-behind before or during the
first layout pass, **name it in XAML**. Reserve `VisualTreeHelper` walks for
elements you genuinely can't name (template parts inside a control template,
items realized by a virtualizing panel) and assume those walks can return
null until after the first `LayoutUpdated`.
